# Token Optimizer

> **核心目标：** 在 Trae IDE 中最大化每 token 的产出价值，重点实现**自动上下文压缩**，避免浪费上下文窗口和 API 费用。

## 核心：自动上下文压缩

> 这是本技能的**首要职责** — 任何操作前，先判断上下文是否需要压缩。

### 0.1 强制压缩阈值

| 阈值条件 | 必须执行的操作 |
|---------|--------------|
| **对话 ≥ 8 轮** | 立即生成 `<summary>` 摘要（即使任务未完成） |
| **单文件修改 ≥ 5 处** | 压缩为批量变更描述，移除旧版本记录 |
| **即将调用高消耗工具前** | 先压缩当前上下文再执行 |
| **用户输入"继续"/"接上"/"接着"** | 先输出简要回顾锚点 |
| **切换到不相关任务** | 强制总结前任务并清空工作记忆 |
| **上下文接近模型限制** | 删除最旧对话，只保留最近 3 轮 + 摘要 |

> 压缩后形成「摘要 + 最新 3 轮 + 当前任务」的三段式结构：

```
[摘要块] ← 历史压缩结果
[最新3轮] ← 最近的关键交互
[当前任务] ← 正在执行的任务状态
```

### 0.2 自动摘要格式

每次压缩使用标准格式：

```
<summary type="task">
  任务：[当前任务名称]
  进度：[已完成 / N/M]
  关键决策：[核心决定]
  文件变更：[变更文件列表]
  下一目标：[下一步]
</summary>
```

### 0.3 增量压缩机制

- 每次新压缩**只追加增量信息**，不重新生成全部内容
- 关键决策链做摘要（保留因果，删除实现细节）
- 代码变更只保留「变更了什么 + 行号范围」，不保留旧代码原文

### 0.4 上下文健康检查流程

每轮对话响应前快速自检：

```
1. 当前对话轮次是否 ≥ 8？           → 是 → 压缩
2. 当前有效 token 是否接近限制？      → 是 → 清理 + 压缩
3. 本次响应预计产生多少新内容？       → 大 → 先压缩再响应
4. 是否有不需要的历史片段可以被归档？ → 是 → 移出活跃上下文
```

## 一、上下文压缩策略（详细）

### 1.1 对话压缩时机

当以下情况发生时，主动对当前对话进行压缩汇总：

| 触发条件 | 操作 |
|---------|------|
| 对话超过 10 轮交互 | 用 `<summary>` 标签汇总关键进展 |
| 单个文件修改超过 5 处 | 压缩为批量变更描述 |
| 进入新任务阶段 | 先总结已完成工作，再开始新任务 |
| 用户输入"继续"/"接上" | 提供简要回顾锚点 |

### 1.2 压缩格式

```
<summary type="task">
  当前任务：[任务名称]
  已完成：[已完成步骤]
  进行中：[当前步骤]
  关键决策：[做出的重要决定]
  文件变更：[变更的文件列表]
</summary>
```

## 二、文件读取优化

### 2.1 读取优先级

```
优先使用（低 token 消耗）：
  Grep          → 搜索特定模式/关键词
  SearchCodebase → 语义搜索相关代码
  Glob          → 按文件名查找

次选（中等 token 消耗）：
  Read (行范围)  → 只读需要的行段
  LS            → 列出目录结构

最后选择（高 token 消耗）：
  Read (全文)    → 仅在必要时读取整个文件
```

### 2.2 大文件读取规则

- 超过 300 行的文件：**必须**按行范围分段读取
- 最多同时读取 3 个不相关文件
- 同一文件连续读取间隔至少隔 2 轮对话

## 三、Todo 管理规范

### 3.1 使用时机

- 3 步以上的复杂任务 → 创建 Todo 列表
- 用户明确列出多项需求 → 创建 Todo 列表
- 单一简单任务 → **不创建** Todo 列表（避免浪费）

### 3.2 压缩原则

- Todo 描述控制在 15 字以内
- 同时不超过 10 项
- 完成后立即移除（不保留已完成项）

## 四、日志与输出控制

### 4.1 日志文件

- `logs/` 目录只读取最新文件的尾部（后 50 行）
- 优先使用 `CheckCommandStatus` 的 `filter` 参数过滤关键词
- 长输出使用 `output_character_count` 限制字符数

### 4.2 游戏日志（KubeJS）

## 五、DeepSeek API 成本控制

### 5.1 模型选择策略

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 简单问答/查询 | DeepSeek-V3 | ¥2/百万 tokens，经济 |
| 复杂代码生成 | DeepSeek-V3 | 性价比最优 |
| 深度推理/分析 | DeepSeek-R1 | ¥4/百万 tokens，需时才用 |
| 余额不足 ¥10 | 精简对话，合并请求 | 节省用量 |

### 5.2 请求合并

- 多个独立的查询 → 一次性发送（bulk）
- 文件修改 → 集中编辑，减少轮次
- 搜索任务 → 批量并行搜索

### 5.3 Prompt Cache 优化策略

DeepSeek API 按**前缀匹配**计算缓存折扣。相同前缀的请求，匹配部分按缓存价（约 ¥0.5/百万）计费。

#### 缓存破坏元凶

| 行为 | 后果 |
|------|------|
| 不压缩上下文 | 每轮追加新内容，前缀持续变化 → 缓存全 miss |
| 频繁读不同大文件 | 每次对话塞入大量不同文件内容 → 前缀无重复 |
| V3/R1 频繁切换 | 两模型缓存池独立 → 切换即清空 |
| 多个分散小请求 | 每次前缀不同 → 无法复用前缀缓存 |

#### 优化策略

1. **固定前缀结构**：所有请求统一为 `[摘要块] + [当前任务] + [具体问题]`，确保共享前缀
   - 大文件内容用工具读取（Read/Grep），**不塞入对话上下文**
   - 对话中只保留「文件路径 + 关键行号」，不保留原文
2. **同一任务固定模型**：尽量在单个任务阶段内不切换 V3/R1
3. **批量请求替代串行**：多个无关查询合并为一次请求发送
4. **前缀长度控制**：保持活跃前缀在 4K tokens 以内，最大化缓存命中概率
5. **上下文未压缩时不发起新请求**：检查对话轮次 ≥ 8 时，先压缩再发请求

## 7×24 自动触发规则

以下场景**必须**自动执行优化策略：

1. **文件读取前**：先思考是否需要全文读取，能否用搜索替代
2. **对话超过 8 轮**：生成上下文摘要
3. **用户切换任务**：先总结前任务，清理上下文
4. **调用耗时工具前**：先压缩当前对话再调用
5. **余额不足时**：主动提示并建议精简策略
6. **缓存命中率异常**：连续 3 次请求均未命中缓存 → 检查前缀一致性，强制压缩上下文
7. **跨任务切换并切换模型**：先压缩上下文再切换模型，避免两池同时失效
